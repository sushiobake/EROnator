# プロジェクト方針・設計メモ（ロードマップ＋Q&A）

いつでも取り出して参照できるようにまとめたドキュメント。

**更新日: 2026-02-09**

---

## ① シミュレーション運用

- シミュレーションを回し続ける。失敗したものは繰り返し試行する。
- 何度も失敗するものはチェック → タグ調整 or 質問設計で対応。
- AI は使わないのでクレジットはかからない。体制が整えばかなり回してよい。
- 将来的に「はい/いいえ」以外（たぶんそう・わからない等）をランダムで混ぜて結果を見たい。**急ぎではない。**

---

## ② 細かい修正と残り設計

### ★ 正解画面

- **おすすめを横並び**にする。レイアウト修正。
- **同一作者は並べない**（シリーズのときに他作品が並ぶのを避ける）。
- **最大5件、同一作者は1本まで**。正解できなかったときの top1–5 と同じルールに揃える。
- **「どれくらいこの作品に近いか」を ○○％で表示**。高いほど購買意欲につなげる。
  - 一致度は「低すぎると意味がない」ので、嘘をつかない程度に**ある程度高い数字**にしたい。試してみて調整する。
- 正解画面は将来的に大事（ここで買うかどうかで収入が変わる）ので丁寧に進める。
- 「FANZAで見る」は小さめでよい。
- 文言例:「正解！？やっぱりね！」「そんなあなたには…おすすめもあるわ！」

### ★ 正解できなかったとき

- 最大30問は維持。これ以上増やすのは微妙。
- その時点の**残り top1〜5** を並べる（既に REVEAL で外した作品は候補から外したうえで、確度順の上位5件）。**同一作者は1本まで**に限定する。
- 「うーん…ちょっとわからなかったわ。ちなみにこの中にはある？」
- 「どれくらい近いか」の％表記も検討（上記の「ある程度高い数字」方針に合わせる）。
- **それでも無い場合は作品名を手入力** → 履歴に残す。
  - **候補から選択は使わない**。手入力（雑でよい）で受け取り、**こちら側で後からうまく処理する**。候補にない等のややこしい問題を避ける。
  - 履歴には**生の入力文字列**をそのまま保存。作品の取り込み・workId 紐づけは後で人が見て判断する。正確なタイトルでない可能性もあるため。
  - ここまで入力してくれるケースは非常にまれだと思い、データ量は少ないので扱いは難しくない想定。

### ★ 本番用

- **すべて履歴を取る**。貴重なデータになる。
- **履歴の形**: **1プレイ = 1レコード**で、その中に「質問列・回答列・結果・時刻」などをまとめて持つ形で十分。
- 有名度は当面使わない。DB が 1 万件くらいになったら使う想定。
- **有名度加算**は、履歴さえ取れていれば実装はそれほど難しくない。プレイから加算できるように設計しておく（後から有効化するだけ）。

### ★ 推薦モード

- **「近日実装」**でよい。急ぎではない。
- 「あなたが妄想～」→「あなたが欲しいのは・・・」などに変更予定。
- ソフト/ハードコンフィムは要らない。前半はまとめ質問で絞り、後半（6問目・11問目など）で選択肢を変える。
- 推薦モードとゲーム正解画面で「同じ確度／類似度計算を流用」するまではやらない。ゲームで正解した人にはその場でおすすめを出しているので、そこまでの一貫性は不要。
- 詳細な仕様（案1: 4タグ＋どれも要らない / 案2: 2選択＋どちらでもない 等）はイメージ固まっているため、ここではスルー。あとから実装でよい。

### ★ 稲荷さんモード・「どちらでもいい」

- 「どちらでもいい」を5回選んだときの**稲荷さんモード**（稲荷さんの漫画を紹介するギミック、特殊モード）は入れたい。
- 稲荷さんは**意外とすぐできる気がする**ので、余裕があったらやる。
- 推薦モード・稲荷さんモードの詳細は、イメージが固まっているためいったんスルー。

## ③ UI

- 最低限は整っている。優先するなら**フォント**（質問文・答えを太くはっきり）。
- 背景はもう少しちゃんとしたものにしたい（どこで作るかは未定）。
- 稲荷さんは最低でも差分を5枚程度。理想は質問ごとに差分（エロい質問のときは恥ずかしそうな顔など）。β版では差分なし or ランダムでよい。将来的に Vtuber のような表現も検討。

---

## ④ DB

- 100件はほぼタグ完璧、500件は手動でそこそこ、500件は別の旧AIで精度が怪しい。
- DB を増やしつつ**タグの精度も上げる**。β公開後に本格的に取り組みたい。

## ⑤ 公開・宣伝・SNS

### β版の現在地（2026年2月時点）

- **導線**: 質問→回答→正解→おすすめ（失敗時は候補選択 or 作品名手入力）まで一通り動作。テストしながら履歴を集めて調整・DB を回す段階まで来ている。
- **DB**: 約1000件。クオリティはまだ伸ばせるが、悪くはない。シミュレーション（1000作品×5回）で正答率92％（ノイズ0）。5回とも失敗した78作品はタグ要確認候補としてリスト化済み。
- **機能**: 推薦モード・稲荷さんモードは未実装。「近日実装」と表記している。
- **UI**: 最低限。スマホでもプレイ可能なところまで対応。タイトルロゴ未着手。キャラ・背景の追加は後からでも可能。
- **SNS**: 作者 note はほんの少し。X の専用アカウントはまだ。

### 公開のタイミングと「育て方」

- **方針**: 「育ててから公開」より **「公開しながら育てる」** を採用する。
  - 現状でも本番に近い形でプレイ可能。履歴が増えないと離脱・失敗の傾向が分からないため、早めに少しでも本番トラフィックを流す。
- **公開日**: 本番履歴取得と致命的バグがなければ、**日付を決めてその日に URL を出してよい**。SNS は「公開前に完璧に揃える」必要はない。
- **SNS の進め方**:
  - **公開と同時**: X 専用アカウント作成 + 1本目「β公開しました。〇〇ができるゲームです。URL」程度でよい。
  - **公開後**: 細く長く。週1回など、続けられるペースで。プレイしてくれた人の反応や気づきをそのまま投稿の種にできる。
- **育てるの中身**:
  - **プロダクト**: テスト・履歴分析・タグ／DB 調整。公開後も継続。
  - **認知**: SNS・note を無理のないペースで続ける。
  - **機能拡張**: 推薦モード・稲荷さんモードは「近日実装」のまま、公開後に段階的に実装。

### ロードマップ（時期感の目安）

| 時期 | やること |
|------|----------|
| **公開まで 1〜2週間** | 本番で履歴が確実に取れることの確認、致命的バグの有無、表記（利用規約・FANZA リンク等）の最終確認。公開日を 1 日決める。 |
| **公開日** | URL を載せた投稿を X（専用アカウント）と note で 1 本ずつ。知り合いや興味ありそうなコミュニティに「出した」と伝える程度でよい。 |
| **公開後 2〜4週間** | 履歴で離脱・失敗の傾向を把握。失敗の多い作品のタグを少しずつ修正。SNS は続けられるペースで。 |
| **1〜2ヶ月後** | 履歴とネタが溜まってきたタイミングで、推薦モード・稲荷さんモードの実装順を決める。「βで分かったこと」を note や X で 1 本まとめると後からも使いやすい。 |

## 優先度の目安

### 公開前にやること（必須に近い）

- **本番で履歴が確実に取れること**  
  1プレイ＝1レコードで、質問列・回答・結果・時刻が保存されているか。ここが無いと「公開して育てる」の意味が薄いため最優先。
- **致命的バグの有無**  
  スマホでプレイ不能・真っ白にならないか。修正するボタン周りなど、直した部分の軽い回帰確認。
- **表記の整理**  
  推薦モードは「近日実装」のまま。利用規約・免責・FANZA リンク周りは公開前に一度確認（既に済んでいればスキップ可）。

### 公開直後〜数週間で回すとよいこと

- **履歴の確認**  
  離脱が多い画面・失敗が多い作品がないか。シミュで出た「5回とも失敗した作品」と本番の失敗作品の重なりを見る。
- **タグ・DB の軽い改善**  
  失敗が集中している作品からタグ追加・修正。β公開後に本格的に取り組む方針に沿う。
- **SNS の種をためる**  
  プレイしてくれた人の反応・自分が気づいたポイントを、X や note の 1 行メモやストック用メモに残しておく。

### 余裕が出てからでよいもの

- **推薦モード**  
  「近日実装」のまま公開し、履歴と反応を見てから設計を固める。
- **稲荷さんモード**（「どちらでもいい」5回で発動）  
  差別化にはなるが、公開の有無には直結しない。余裕があれば実装。
- **タイトルロゴ・キャラや背景の追加**  
  後から差し替え可能なため、公開を待たず「ある程度できたら差し替え」でよい。

### 優先度一覧（参照用）

| 優先度 | 内容 |
|--------|------|
| **高（公開前）** | 本番の履歴取得、致命的バグ解消、表記の最終確認。正解画面（レイアウト・同一作者除外・％・文言）、正解できなかったとき（top1–5・同一作者1本・手入力・履歴）は既に対応済みなら維持のみ。 |
| **高（公開後）** | 履歴分析、失敗の多い作品のタグ改善。有名度加算の設計（履歴取れていれば後から有効化可能）。 |
| **中** | シミュの失敗分析・タグ修正（5回とも失敗した78作品など）、稲荷さんモード（余裕があれば）。 |
| **低〜後** | 推薦モード実装、UI（フォント・背景・稲荷さん差分）、ノイズ以外の回答バリエーション。 |
| **並行** | β公開と少数ユーザー獲得、SNS・note を細く長く続ける。 |
